# 公開準備リファクタリング 設計 (2026-07-02)

主要機能が一段落した cc-studio を GitHub で一般公開するための、リファクタリングと公開準備の全体設計。

## ゴール

**このリポジトリを履歴ごと GitHub の public リポジトリとして公開する。** 公開前にコード品質リファクタと国際化（日英対応）を完了させる。

## 決定事項

| 論点 | 決定 |
|---|---|
| 公開形態 | このリポジトリ自体を履歴ごと公開（ミラー分離・履歴書き換えはしない） |
| 公開とリファクタの順序 | リファクタを終えてから公開 |
| docs の開発メモ類 | そのまま含める（作業記録も公開） |
| LICENSE | MIT |
| README | 日本語 README.md + 英語 README.en.md の両方を用意 |
| i18n の範囲 | アプリ全体（プラグイン説明・@setting ラベル・UI 文字列・通知） |
| 表示言語の選択 | 端末言語に追従 + アプリ内設定で上書き（端末に合わせる / 日本語 / English） |
| ドキュメント置き場 | `docs/specs`（設計）+ `docs/plans`（実装計画）に統一。`docs/superpowers/` は廃止 |

## フェーズ構成

### フェーズ 1: コード品質リファクタ

1. **MainActivity.kt（1,142 行）の分割** — Kotlin 全体の半分を占める最大のホットスポット。責務ごと（WebView 設定、ダウンロード処理、通知ルーティング、スクリーン切替 UI、SAF 添付など）に協調クラスへ抽出し、抽出したロジックにユニットテストを付ける。分割の具体的な切り方は実装計画で確定する。
2. **プラグイン / JS 側の整理** — 8 本のプラグインと bootstrap.js の共通パターン（設定読み込み・ログ・DOM 監視・フレーム間橋渡し）の規約を揃える。
3. **テスト拡充** — 既存テスト（11 ファイル）の流儀に沿って、Activity に埋まっていたロジックをテスト可能な純関数・小クラスに寄せる。

### フェーズ 2: i18n + プラグイン説明刷新

1. **プラグイン説明の日英対応** — メタヘッダを userscript の慣例に合わせてロケール接尾辞対応に拡張する: `@description`（英語・既定）+ `@description:ja`（日本語）。`PluginMetaParser` が表示言語に応じて選択し、無い言語は英語にフォールバック。`@setting` のラベルも同じ仕組みで日英対応させる（具体形式は実装計画で決定）。
2. **説明文の全面書き直し（8 本）** — 現状の長文説明はプラグインカードの見栄えを崩している。全プラグインを次の定型で 1〜2 文に書き直す:
   - 不具合解消型: 「素の code-server / Claude Code 拡張には〜という不具合（摩擦）がある。このプラグインはそれを〜で解消する」
   - 機能追加型: 「元々存在しない〜の機能を、このプラグインが実現する」
   - README の「問題 → 答え」対処表と言い回しを揃える。
3. **アプリ全体の文字列リソース化** — ハードコードされた日本語 UI 文字列（スクリーン一覧・switcher・ダイアログ・通知・設定画面・プラグインカード）を `values/strings.xml`（英語）+ `values-ja/strings.xml`（日本語）に抽出。言語切替は Android 標準の per-app language（`AppCompatDelegate.setApplicationLocales`）に乗る。

フェーズ 1 の分割で文字列の置き場所が確定してから抽出するため、この順序とする。

### フェーズ 3: 公開体裁

- LICENSE (MIT) を追加。
- README: 現行日本語版を維持しつつ英語版 README.en.md を作成し相互リンク。
- **docs 置き場の統一**: `docs/superpowers/specs/` の 3 本 → `docs/specs/`、`docs/superpowers/plans/` の 2 本 → `docs/plans/` へ移動して `docs/superpowers/` を削除。`docs/specs/` に混入している `*-plan.md` 3 本（observer 系）も `docs/plans/` へ移動。移動後にリポジトリ内の相互参照リンクを grep して更新。
- 個人情報スキャンを最終ガードとして再実行。

### フェーズ 4: 公開

- GitHub に public リポジトリを作成し、履歴ごと push。

## 実装計画時に決める細部

- keyboard-suppress の残課題（#2 自動フォーカス抑制）を先に片付けるか、凍結してリファクタ後に戻るか。
- `@setting` ラベルの i18n 形式（`@setting:ja` の行を足すか、ラベル部の埋め込み記法か）。
- 通知チャンネル名・常駐通知テキストの翻訳の切替点（チャンネルは作成後に改名不可の制約あり）。
- プラグイン管理画面のカード描画が説明文を読む箇所と、言語切替のライブ反映の要否。

## 進め方

フェーズごとに writing-plans で実装計画を作り、レビュー後に実装する。作業は `refactor/public-release` ブランチで行う。他セッションと並行することがあるため、各フェーズ着手時に main との差分を確認して前提を再検証する。
